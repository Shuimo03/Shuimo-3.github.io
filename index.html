Go内存管理

Go内存分配器核心组件

  • heapArena：保存整个虚拟地址空间。
  • mheap：分配的堆，在页大小为8KB的粒度上进行管理。
  • mspan：是mheap上管理的一连串的页
  • mcentral：收集了给定大小等级的所有span
  • mcache：为per-P的缓存

其中页是向操作系统申请内存的最小单位，目前设计为8KB。

内存分配器

内存中有两个重要区域：

  • 栈区域
  • 堆区域

程序中的数据和变量都会被分配到程序所在的虚拟内存中，函数调用，返回值以及局部变量大部分都会被分配在栈上，这一部分内存由编译器管理。

传统意义上的栈被Go的运行时霸占，不开发给用户态代码，而传统意义上的堆内存被Go运行时划分为两个部分：

  • 一部分是被Go运行时自身所需要的堆内存，被成功为堆外内存。
  • 还有一部分是用于Go用户态代码所在的堆内存，被称为Go堆。

Go程序的执行是基于goroutine的，goroutine和普通程序一样都有栈和堆的概念，但是Go的运行时屏蔽掉了这两个概念，只有在运行时区域的时候分别对应goroutine执行栈和Go堆。

当函数返回的时候，在栈上就会被回收，栈中的对象都会被回收，不需要GC标记，但是堆比较麻烦一些，因为Go支持垃圾回收，只要对象在堆上生存，Go 的运行时 GC 就会在 后台将对应的内存进行标记从而能够在垃圾回收的时候将对应的内存回收，进而增加了开销。

内存分配器设计原理

内存管理一般分为以下三种不同的组件：

  • 用户程序(Mutator)
  • 分配器(Allocator)
  • 收集器(Collector)

当用户程序申请内存的时候，它会通过分配器来申请新内存，而分配器会从堆中初始化相应的内存区域。

分配方法

分配方法一般是两种：

  • 线性分配器
  • 空闲链表分配器

线性分配器(Bump Allocator)

线性分配法虽然是一种高效的分配方法，但是有较大的局限性，使用线性分配器的时候，只要在内存中维护一个指向内存特定位置的指针，如果用户程序向分配器申请内存的话，分配器只需要检查剩余的空闲内存，返回分配的内存区域并且修改指针所在的内存位置。

这样做的好处是执行速度较快并且复杂度也低，但是缺点在 如果分配过的内存被回收的话，线性分配器没有办法重新利用起来。

和线性复杂性相搭配的算法有：

  • 标记压缩(Mark-Compact)
  • 复制回收(Copying GC)
  • 分代回收(Generational GC)

空闲链表分配器(Free-List Allocator)

因为不同的内存块通过指针构成了链表，所以使用这种方法的分配器可以重新使用回收的资源，但是因为分配内存的时候需要遍历链表，所以它的时间复杂度是O(n)。空闲链表分配器可以选择不同的策略在链表中的内存块中进行选择，常见的如下：

  • 首次适应(First-Fit): 从链表头开始遍历，选择第一个大小大于申请内存的内存块。
  • 循环首次适应(Next-Fit): 从上次遍历的结束位置开始遍历，选择第一个大小大于申请内存的内存块。
  • 最优适应(Best-Fit)：从链表头遍历整个链表，选择最合适的内存块。
  • 隔离适应(Segregated-Fit)： 将内存分割成多个链表，每个链表中的内存块大小相同，申请内存时先找到满足条件的链表，再从链表中选择合适的内存块；

分级分配

线程缓存分配(Thread-Caching Malloc，TCMalloc)是用来分配内存的机制，它比glibc中的malloc还要快，Go的内存分配器就是借鉴了这种方式，它的核心是使用多级缓存将对象根据大小分类，并且按照类别实施不同的分类策略。

类别 大小
微对象 (0, 16B)
小对象 [16B, 32KB]
大对象 (32KB, +∞)

多级缓存

内存分配器除了会区别大小不同的对象之外，还会把内存分成不同的级别分别管理，TCMalloc和Go运行时分配器都会引入以下三个组件:

  • 线程缓存(Thread Cache)
  • 中心缓存(Central Cache)
  • 页堆(Page Heap)

线程缓存属于每一个独立的线程，它能满足线程上绝大多数的内存分配需求，因为不会涉及到多线程，所以也不会考虑到使用互斥锁来保护内存，这也可以减少锁竞争带来的性能损耗。

当线程缓存不能满足需求的时候，运行时就会使用中心缓存来作为补充解决小对象的内存分配，遇到32k以上的对象时，内存分配器就会选择页堆直接分配大内存。

虚拟内存布局

  • 线性内存
  • 稀疏内存

在1.10以及之前的版本中，Go使用线性内存，在1.10之后Go使用了稀疏的堆内存来替代了连续的线性内存，解决了内存带来的限制以及在特殊情况下可能出现的问题。

线性内存

Go在启动的时候会虚拟化整个内存区域，如下三个区域：

  • spans：存放着指向内存管理单元runtime.mspan指针，每个内存单元都会管理几页的内存空间，每页大小为8KB。
  • bitmap：用来标识arena区域中那些地址保存了对象，位图中的每个字节都会表示堆区中的32字节是否空闲。
  • arena：区域是真正的堆区，运行时会把8KB看作一页，这些内存中存储了所有在堆上初始化的对象。

分别预留了512MB,16GB以及512GB的内存空间，这些内存空间并不是真正的物理内存，而是虚拟内存。

对于任意一个地址，我们都可以根据arena的基地址计算出该地址所在的页数并且通过spans数组获得管理该片内存的管理单元，spans数组中多个连续的位置可能都对应着一个runtime.mspan结构。

线性内存的不足

Go会在垃圾回收的时候根据指针的地址判断对象是否在堆中，并通过之前介绍的过程找到管理该对象的runtime.mspan，这些都是建立在堆区内存是连续的假设上，这种方式虽然简单，但是在C和Go混合使用的时候就会导致程序崩溃。

  • 分配的内存地址会发送冲突，导致堆的初始化和扩容失败。
  • 没有被预留的大块内存可能会被分配给 C 语言的二进制，导致扩容后的堆不连续。

线性的堆内存需要预留大块的内存空间，但是申请大块的内存空间而不使用是不切实际的，不预留内存空间却会在特殊场景下造成程序崩溃。

稀疏内存

使用稀疏内存布局可以移除堆大小的上线，还可以解决C和Go混合使用的地址空间冲突问题，不过因为时稀疏内存管理，所以就会失去内存的连续性，这样就会导致内存管理变的复杂。

运行时使用二维数组管理所有的内存，每个单元都会管理64MB。

地址空间

因为所有的内存最终都要从操作系统中申请，所以Go的运行时构造了操作系统的内存管理抽象层，该抽象层在运行时管理的内存空间分为以下四种状态：

状态 解释
None 内存没有被保留或者映射，是地址空间默认的状态。
Reserved 运行时持有该地址空间，但是访问该内存会导致错误。
Prepared 内存被保留，一般没有对应的物理内存访问该片内存的行为是未定义的可以快速转换到 Ready 状态
Ready 可以被安全访问

运行时使用Linux提供的mmap，munmap和madvise等系统调用实现了操作系统的内存管理抽象层，抹平了不同操作系统的差异，为运行时提供了更加方便的接口，除了 Linux 之外，运行时还实现了 BSD、Darwin、Plan9 以及 Windows 等平台上抽象层。

内存管理组件

  • 内存管理单元
  • 线程缓存
  • 中心缓存
  • 页堆

内存分配

堆上所有的对象都是通过调用runtime.newobject函数分配内存的，该函数会调用runtime.mallocgc来分配指定大小的内存，这也是用户程序在堆上申请内存空间的必经函数。

对象大小

对象大小可以分成三种：

  • 微对象(0,16B)：先使用微型分配器，再依次尝试线程缓存、中心缓存和堆分配内存；
  • 小对象([16KB,32KB])： 依次尝试使用线程缓存、中心缓存和堆分配内存；
  • 大对象(32KB, +∞)：直接在堆上分配内存；

微对象

Go运行时将小于16字节的对象划分微对象，它会使用线程缓存上的微分配器来提高微对象分配的性能，主要使用它来分配较小的字符串以及逃逸的临时变量。微分配器可以将多个较小的内存分配请求合入同一个内存块中，只有当内存块中的所有对象都需要被回收时，整片内存才可能被回收。

小对象

小对象是指大小为16字节到32字节的对象，以及所有小于16字节的指针类型对象，小对象的分配可以被分成以下三个步骤：

  • 确定分配对象的大小以及跨度类runtime.spanClass。
  • 从线程缓存，中心缓存或者堆中获取内存管理单元并从内存管理单元找到空闲的内存空间。
  • 调用runtime.memclrNoHeapPointers清空空闲内存中的所有数据。

大对象

运行时对于大于32KB的大对象进行单独处理，不会从线程缓存中或者中心缓存中获取内存管理单元，而是直接使用runtime.mcache.allocLarge分配大片内存，也就是直接从Go堆上进行分配。

runtime.mcache.allocLarge会计算分配该对象所需要的页数，它按照8KB的倍数在堆上申请内存。

Java集合总结

集合概要

Java的集合主要包括Collection和Map两种，Collection存储对象的集合，Map存储键值对的映射表。

Set

  • TreeSet：基于红黑树实现，支持有序性操作，比如根据一个范围查找元素操作，但是查找效率不如HashSet，HashSet的查找时间是O(1)，TreeSet是O(logN)。
  • HashSet：基于哈希表实现，支持快速查找，但不支持有序性操作，并且失去了元素插入顺序的信息，也就说是不稳定性的，使用Iterator遍历HashSet所得到的结果是不稳定的。
  • LinkedHashSet：具有HashSet的查找效率，并且内部使用双链表维护元素的插入顺序，也就是说是一个稳定版本的HashSet。

List

  • ArrayList：基于动态数组实现，底层使用object数组实现，支持随机访问。
  • Vector：和ArrayList类似，但是它是线程安全的。
  • LinkedList：基于双向链表实现，只能顺序访问，但是可以在O(1)时间内插入，还可以作为栈，队列和双向队列。

ArrayList和Vector的对比

  • Vector是同步的，所以开销要比ArrayList要大，访问速度更慢，所以最好使用ArrayList，同步操作完全可以自己来做。
  • Vector每次扩容请求都是自生的2倍，虽然可以通过构造函数来设置增长的容量，而ArrayList则是1.5倍。

线程安全的ArrayList版本：

  • Collections.synchronizedList;
  • CopyOnWriteArrayList

LinkedList和ArrayList比较

ArrayList基于动态数组，LinkedList基于双向链表实现，ArrayList和LinkedList本质上可以归根据数组和链表的区别：

  • 数组支持随机访问，但是插入删除的代价很高，需要移动大量元素。
  • 链表不支持随机访问，但是插入删除只需要改变指针。

Queue

  • LinkedList：可以使用它来实现双向队列。
  • PriorityQueue：基于堆结构实现，可以使用它实现优先队列。

Map

  • TreeMap：基于红黑树实现。
  • HashMap：基于哈希表实现。
  • HashTable：和HashMap类似，但是HashTable是线程安全的，也就是说多个线程写入HashTable 不会导致数据不一致。它是遗留类，不应该去使用它，而是使用 ConcurrentHashMap 来支持线程安全，ConcurrentHashMap 的效率会更高，因为 ConcurrentHashMap 引入了分段锁。
  • LinkedHashMap：使用双向链表来维护元素的顺序，顺序为插入顺序或者最近最少使用（LRU）顺序。

集合底层实现

List

  • ArrayList: Object[]数组
  • Vector：Object[]数组
  • LinkedList：双向链表(JDK1.6之前是循环链表，JDK1.7之后取消了循环)

Set

  • HashSet(无序,唯一)：基于HashMap实现的，底层使用HashMap来保存元素。
  • LinkedHashSet：LinkedHashSet是HashSet的子类，并且内部使用LinkedHashMap。
  • TreeSet(有序，唯一)：红黑树实现。

Map

HashMap：JDK1.8之前主要是数组和链表组成，数组是HashMap的主题，链表是为了解决哈希冲突而存在的，因为HashMap默认使用了拉链法，但是在1.8的时候就加入红黑树，当链表长度大于8并且数组长度大于64的时候，就会从链表转换为红黑树。

HashMap的特点：

  • 支持null作为键值对。
  • 无法保证键值对顺序。
  • HashMap是不安全的，如果想要使用线程安全的hashMap应该使用concurrenthashmap。

如果只是链表长度大于8，数组长度小于64，那不会转换为红黑树，会先对数组进行扩容。

LinkedHashMap：LinkedHashMap继承HashMap，所以它的底层依然是HashMap这一套，除此之外，LinkedHashMap还增加了一条双向链表，保证了键值对的插入顺序。

HashTable：数组+链表组成，数组是HashTable的主体，链表主要是解决哈希冲突，它是线程安全的，因为内部使用了sync。

集合选型

主要根据集合的特点来选用，比如我们需要根据键值来获取元素时就是用Map接口下的集合，需要排序的时候就是用TreeMap，不需要排序就使用HashMap，需要保证线程安全的话就是用ConcurrentHashMap。

只需要存放元素值时，就选择实现Collection 接口的集合，需要保证元素唯一时选择实现 Set 接口的集合比如 TreeSet 或 HashSet，不需要就选择实现 List 接口的比如 ArrayList 或 LinkedList，然后再根据实现这些接口的集合的特点来选用。

ArrayList,Vector,LinkedList

ArrayList是一个动态数组，它的默认大小是10，每次扩容的倍数都是1.5倍，既然是动态数组，那它的本质就是数组，所以它的特性如下：

  • 查找速度速度快：O(1)
  • 修改速度(插入删除)慢：O(n)

从而可以看出ArrayList适合频繁的查找工作，但是它是线程不安全的，如果想要线程安全的话，就可以使用以下集合：

  • CopyOnWriteArrayList
  • Collections.synchronizedList

给add和remove上锁。

Arraylist和Vector的区别

线程安全方面：

  • Arraylist是线程不安全的
  • Vector是线程安全的

扩容方面：

  • Arraylist扩容是1.5倍。
  • Vector是两倍。

ArrayList和LinkedList的区别

线程安全方面：两者都是线程不安全的，底层实现方面：

  • LinkedList是双向链表
  • ArrayList是动态数组

所以两者除了线程安全上的区别，还有就是数据结构上的区别。

  • 数组对于链表来说，查找速度快，但是插入删除操作代价高。
  • 链表虽然查找效率不行，但是因为物理上可以不用连续，逻辑上连续的原因，它的删除操作快。

但是也要分情况，如果ArrayList使用的是add(E e)方法，就会默认的在数组尾部添加元素，这个时候就是O(1)，但如果是指定的话就是O(n-i)。因为在进行上述操作的时候集合中第 i 和第 i 个元素之后的(n-i)个元素都要执行向后位/向前移一位的操作。

内存空间上 LinkedList比ArrayList更耗内存，因为LinkedList需要存储前驱节点和后继节点，而ArrayList 主要体现在list列表的结尾会预留一些容量空间。

ArrayList扩容机制

ArrayList有三种方式来初始化，它默认使用了无参构造方法，因为Java8使用了延迟初始化，所以并不会马上的创建长度为10的数组，而是在调用add方法的时候，才会对elementData初始化。

第二种是指定构造方法：传入初始容量initialCapacity，如果初始容量大于0，就会直接创建一个指定大小的object数组，如果初始容量等于0，elementData就会指向共享的空数组EMPTY_ELEMENTDATA，如果小于0的话就会抛出异常IllegalArgumentException 。

int newCapacity = oldCapacity + (oldCapacity >> 1); 这是扩容的倍数，所以每次扩容之后容量都会变成原来的1.5倍，如果oldCapacity是偶数就是1.5倍，否则的话就是1.5倍左右。

  • 将原来数组长度(elementData.length)赋值给oldCapacity。
  • oldCapacity扩容完之后赋值给newCapacity。
  • 检查newCapacity是不是大于最小需要容量，如果还是小于最小需要容量，就会把最小需要容量作为数组新容量。
  • 如果新容量大于 MAX_ARRAY_SIZE,进入(执行) hugeCapacity() 方法来比较 minCapacity 和 MAX_ARRAY_SIZE。

无序性与不可重复性

无序性不等于随机性，无序性是指存储的数据在底层数组中并非按照数组索引的顺序添加，而是根据数据的哈希值来决定的。

不可重复性是指添加的元素按照equals()判断时返回false，需要同时重写equals方法和hashcode方法。

HashSet,LinkedHashSet和TreeSet

  • HashSet是Set接口的主要实现类，HashSet 的底层是 HashMap，线程不安全的，可以存储 null 值；
  • LinkedHashSet 是 HashSet 的子类，能够按照添加的顺序遍历；
  • TreeSet 底层使用红黑树，能够按照添加元素的顺序进行遍历，排序的方式有自然排序和定制排序。

Map接口

HashMap和HashTable的区别

线程方面来看，HashMap是线程不安全的，而HashTable是线程安全的，因为HashTable的内部方法基本都使用过了synchronized修饰，如果想要使用线程安全的HashMap就可以使用ConcurrentHashMap。

效率方面，因为HashTable使用了锁，所以HashMap要比HashTable的效率高一点，另外HashTable基本上被淘汰了。

HashMap是可以存储null的键值对的，但是null作为键只能有一个，null作为值可以有很多个，而HashTable不可以。

初始化容量大小和每次扩容大小是不一样的，HashTable默认大小是11，之后每次扩容是原来的2n+1。

HashMap默认的是16，之后每次扩容，容量变为原来的2倍，如果在初始化的时候给定容量初始值，那HashTable会直接使用给定的大小，而HashMap会将其扩充为2的幂次方大小。

HashMap put流程

  • 判断键值对数组table[i]是否为空或者null，如果是的话执行resize()进行扩容。
  • 根据键值key计算hash值来得到插入数组索引i，如果table[i]==null，判断实际存在的键值对数量size是否超多了最大容量threshold，如果超过，进行扩容。
  • 如果如果table[i]不为空，就判断table[i]的首个元素是否和key一样，如果相同直接覆盖value，这里的相同是指equals和hashcode相同。
  • 如果不相同的话，判断table[i] 是否为treeNode，即table[i] 是否是红黑树，如果是红黑树，则直接在树中插入键值对。
  • 遍历table[i]，判断链表长度是否大于8，大于8的话把链表转换为红黑树，在红黑树中执行插入操作，否则进行链表的插入操作；遍历过程中若发现key已经存在直接覆盖value即可。

HashMap和HashSet的区别

HashSet底层就是基于HashMap实现的，除了clone()、writeObject()、readObject()是 HashSet实现的，其他都是直接调用HashMap中的方法。

HashMap HashSet
实现了 Map 接口 实现 Set 接口
存储键值对 仅存储对象
调用 put()向 map 中添加元素 调用 add()方法向 Set 中添加元素
HashMap 使用键（Key）计算 hashcode HashSet 使用成员对象来计算 hashcode 值，对于两个对象来说 hashcode 可能相同，所以equals()方法用来判断对象的相等性

HashMap底层实现

JDK1.8之前HashMap底层是 数组+链表组合而成的链表散列，HashMap通过key的hashCode通过哈希函数处理后得到hash值，然后通过 (n - 1) & hash 判断当前元素存放的位置（这里的 n 指的是数组的长度），如果当前位置有元素的话，就会判断这个元素是不是和要存入的元素一样，引用的话就会直接覆盖，不相同的话就会通过拉链法来解决冲突。

JDK1.8之后解决哈希冲突有了较大的变化，当链表长度大于8并且数组长度大于64的时候就会从链表转换成红黑树，选择红黑树的原因也很简单，因为红黑树适合大量插入删除并且时间复杂度都能控制在O(nlogn)。

如果只是链表长度大于8但是数组长度小于64的话，那会对数组进行扩容而不会进行转换。

拉链法

因为HashMap的本质就是散列表，而散列表可以看成是数组的一种变形，为了解决哈希冲突就需要在数组中每个位置上添加一个链表，如果碰到哈希冲突的话，就把冲突的值添加到链表中就可以了。

ConcurrentHashMap实现

这是1.7的ConcurrentHashMap，底层使用了分段数组+链表实现，对每个数组进行了分割分段，每一把锁只锁容器其中一部分数据，当多线程访问容器里面不同数据段的数据，就不会存在锁竞争问题，提高了并发效率。

到了1.8改成了使用synchronized +CAS操作，整个看起来像是一个线程安全的HashMap，虽然在 JDK1.8 中还能看到 Segment 的数据结构，但是已经简化了属性，只是为了兼容旧版本。

程序计数器 PC

程序计数器是一块比较小的内存空间，存放着下一条指令所在的单元地址，字节码解释器在工作的时候就是通过改变计数器的值来选取下一条需要执行的字节码指针，像分支，循环，跳转，异常处理，线程恢复等等都需要使用这个计数器来完成。

每个线程都具有一个独立的程序计数器，因为这样的话可以保证线程切换后能恢复 到正确的执行位置。

如果线程执行的是一个Java方法，这个计数器记录的就是正在执行的虚拟机字节码指令地址。

如果是Native方法，计数器就为空，同时PC也是唯一一个没有OutOfMemoryError异常的区域。

Java虚拟机栈

Java虚拟机栈也是线程私有的，它的声明周期和线程一样，每个Java方法都会在执行的时候创建一个栈帧用来保存局部变量表，操作数栈，常量池引用等信息，从方法调用直到执行完成的过程，对应着一个栈帧在Java虚拟机栈中的入栈和出栈过程。

可以通过-Xss参数来指定每个线程的Java虚拟机栈内存大小，如果是32位Java App，那默认的栈大小是320K，如果是64KB就是1024K。

这个区域可能还会抛出以下异常：

  • 当线程请求的栈深度超过最大值的话，就会抛出StackOverflowError异常。
  • 栈进行动态扩展的时候如果没有办法申请到足够的内存，就会抛出OutOfMemoryError 异常。

Java本地方法栈

本地方法栈类似于Java虚拟机栈，只不过它们的区别是本地方法栈调用的是其他语言(C/C++/汇编)，并且被编译为基于本机硬件和操作系统的程序。

堆

堆是Java虚拟机所管理的内存中最大的一块，所有线程都共享堆的内存区域，所有的对象都是在堆上面来分配内存，是垃圾收集的主要区域，现在的垃圾收集器基本上都是使用分代收集算法，主要的思想是针对不同类型的对象使用不同的垃圾回收算法，可以把堆分成两块。

  • 新生代
  • 老年代

堆不需要连续的物理内存，需要连续的逻辑内存，并且还可以动态增加内存，还是一种先进先出的结构，增加失败的时候就会抛出OutOfMemoryError异常。

方法区

每个线程都可以共享方法区内存区域，用来存放已经被加载的类信息，常量，静态变量还有即时编译器编译之后的代码等数据，它也不需要连续的内存，并且可以动态扩展，动态扩展失败一样会抛出 OutOfMemoryError 异常。

方法区的目标是对常量池的回收和对类的卸载，但是一般很难实现，在HotSpot虚拟机把它当成永久代进行垃圾回收。

运行时常量池

运行时常量池是方法区的一部分，class文件中的常量池(编译器生成的字面量和符号引用)，会被类加载之后放进这个区域。

直接内存

在JDK1.4中引入的NIO类，它可以使用Native函数库直接分配堆外内存，如何通过Java堆里的DirectByteBuffer对象作为这块内存的引用进行操作，这样的话，能在一些场景里面提高性能，因为避免了在堆内存和堆外内存来回拷贝数据。

年轻代

年轻代用来存放最近创建的对象，尺寸随着堆大小的增大和减少从而完成相对应的变化，默认值是保持为堆大小的1/15，可以通过-Xmn参数来设置年轻代为固定大小。

年轻代的特点是对象更新速度块，在短时间内产生大量的死亡对象，并且它的空间是连续可用的，所以使用复制清除算法和并行收集器进行垃圾回收，它所对应的垃圾回收被称为初级回收。

老年代

Full GC 是发生在老年代的垃圾收集动作，所以使用标记-清除算法，标记-清除算法会产生内存碎片，在执行full gc的情况下，会阻塞程序的正常运行，老年代的gc比年轻代的gc效率上慢10倍以上。对效率有很大的影响。

永久代

永久代是hotspot虚拟机，它不属于堆内存，是方法区的一种实现，永久代存放JVM运行时所需要的类，包含java库的类和方法。

对象创建过程

  • 类加载检查
  • 分配内存
  • 初始化零值
  • 设置对象头
  • 执行init方法

类加载检查

当虚拟机碰到一条new指令的时候，会先去检查这个指令参数能不能在常量池中定位到一个类的符号引用，并且检查这个符号引用的类有没有被加载，解释和初始化过，如果没有的话就会进行类加载。

分配内存

对象所需要的内存大小在类加载完成之后就可以完全确定了，为对象分配空间的任务等于把一块确定大小的内存从Java堆里面划分出去，有两种不同的情况：

  • 堆中内存绝对规整的：使用指针碰撞方法。
  • 堆中内存并不是规整的：使用空闲列表

指针碰撞方法是指如果内存是绝对规整的，内存一分为二，假设使用过的内存在右边，没有使用过的在左边，中间放一个指针，只需要把指针指向空间内存区域移动对象所需要的大小相等的距离就可以了。

空闲列表的方法是虚拟机维护一个列表，记录上面那些内存块是可以用的，在分配的时候从列表中找到一块组口大的空间划分给对象实例，然后更新列表上的记录。

内存分配并发问题

因为对象创建在虚拟机中是非常频繁的行为，即使修改了一个指针所指向的位置，在并发的情况下也并不是线程安全的。

比如对象A分配内存，指针还没有修改，对象B又同时使用原来的指针来分配内存。

解决这种问题有两种办法：

  • 对对象分配空间动作进行同步处理，使用CAS加上失败重试的方法保证更新操作的原子性。
  • 把内存分配动作按照线程 划分在不同的空间之中执行，即每个线程在Java堆中预先分配一小块内存，这种方法称为本地线程分配缓冲。

对象的内存布局

  • 对象头
  • 实例数据
  • 对其填充

对象头信息

对象头包括两部分信息，第一部分用来存储对象本身的运行时数据，比如哈希码，GC分代年龄，锁状态标志，线程持有的锁，偏向线程ID等等。

第二部分是类型指针，即对象指向它的类元数据的指针，虚拟机通过这个指针来确定对象是能够类的实例。

HashMap的原理理解

HashMap简介

在了解HashMap之前，我们先要了解哈希表，哈希表也是散列表，它是普通数组的推广，在哈希表中，它是根据键直接来访问内存存储位置的。通过一个计算键值的函数，将要查询的数据映射到表中的一个位置来访问记录，这样可以加快查找速度，这个映射函数就是散列函数，里面存放的记录数组就是散列表。

哈希函数

  • 除法散列法
  • 乘法散列法
  • 全域散列法

解决哈希冲突

哈希冲突是指两个关键字映射到了一个值上面，这样的情况就是哈希冲突，而解决哈希冲突的方法有两种：

  • 链接法
  • 开发寻址法

链接法

对比开放寻址法，链接法要简单的很多，在链接法里面，每个桶或者槽都放在一条链表里面，有可能是单链表也有可能是双链表，双链表的话，删除比较快。

它的插入很快通过哈希函数计算出散列槽位，然后插入到对应的链表中就可以，时间复杂度是O(1)，但是当查找和删除一个元素的时候，还是一样就算出对应的槽，然后遍历链表查找或者删除，它的时间复杂度由链表长度所决定，所以O(K)。

开放寻址法

所有的元素都存放在散列表里面，也就是说每个表项或者包含动态集合的一个元素，或者NIL，当要查找某个元素的时候，会系统的查找所有表项，直到找到所需要的元素，或者该元素不存在，也就查找失败了。

它和链接法不一样的地方在于，它没有链表，也没有元素存放在散列表外，所以在开放寻址法中，散列表可能会被填满，从而导致不能插入任何新的元素。

三种常用的计算开放寻址法的探查序列：

  • 线性探查
  • 二次探查
  • 双重探查

HashMap的实现

HashMap在JDK7中的实现是数组+链表，到了JDK8中则是数组+链表或红黑树，数组是HashMap的主体，而链表是为了解决哈冲突而存在的，当链表的长度大于8并且数组长度大于64的时候，链表就会转换为红黑树，如给只是单纯的链表长度大于8，就不会发生转换，会先对数组进行扩容。

选择红黑树的原因

因为hashMap插入和删除相对频繁的时候，红黑树的效率比较高，如果是读频繁的话就会使用AVL，因为AVL是严格平衡的。

HashMap的源码解释

下面是哈希表的一些基本信息：

HashMap的构造方法

HashMap的构造函数一共有四种：

  • public HashMap()
  • public HashMap(Map<? extends K, ? extends V> m)
  • public HashMap(int initialCapacity) // 指定了表容量的大小。
  • public HashMap(int initialCapacity, float loadFactor)

public HashMap()

如果是没由指定大小的话，就会使用默认大小也就是16。

public HashMap(Map<? extends K, ? extends V> m)

使用和指定的Map相同映射构造一个新的HashMap，加载因子默认还是0.75和初始容量，初始容量足以容纳指定Map中的映射。

public HashMap(int initialCapacity, float loadFactor)

指定容量和加载因子，但是容量不能小于0，要不然就会抛出IllegalArgumentException错误，加载因子也不能为NaN或者小于0。

确定哈希桶数组索引的位置

这个方法是1.7和1.8都有的，不管是增加，删除或者查找键值对，定位到哈希桶数组的位置都是关键的第一步，通过hash算法可以很快的得到对应元素的位置，不用遍历链表，所以hash方法的好坏据欸的那个了HashMap定位数组索引位置的好坏。

这里的hash算法本质上就是三步：

  • 取key的哈希值
  • 高位运算
  • 取模运算

在JDK1.8中的实现中，优化了高位运算的算法，通过hashCode()的高16位异或或者是低16位，

static class Node<K,V> implements Map.Entry<K,V>

这是哈希表的桶节点定义，是HashMap的一个内部类，实现了Map.Entry接口，本质上就是一个映射对，之前哈希表的图中黑点就表示一个Node对象。

get操作

  1. 首先计算key的hash值。
  2. 判断键值对对数组table[i]是是否为空或者null。
  3. 如果获取到的元素的 key 值相等，说明查找到了，返回元素。
  4. 如果获取的元素的key值不相等，查找next结点的元素
    1. 如果是红黑树，就在红黑树中查找。
    2. 不是红黑树，就遍历链表查找，找到就返回。

put操作

简单来说，put的实现一共有六个步骤：

  1. 首先判断table是不是为空，如果为空的话，就是使用reszie()进行扩容。
  2. 根据键值key来计算hash得到插入的数组索引i，如果table[i] == null，直接在新建节点添加，然后进行扩容。
  3. 判断table[i]的第一个元素是不是和key一样，如果相同的话就直接覆盖value，否则话就转到第四步，这里的相同指的是hashCode以及equals。
  4. 判断table[i]是否为treeNode，即table[i]是否为红黑树，如果是红黑树就在树中插入键值对。
  5. 遍历table[i]，判断链表长度是否大于8，大于8的话把链表转换为红黑树，在红黑树中执行插入操作，否则进行链表的插入操作，在遍历过程中如果发现key已经存在了，那直接覆盖value就可以了。
  6. 插入成功之后，判断实际存在的键值对size是否超过最大容量threshold，超过的话就需要扩容。

resize()扩容方法

在HashMap中，桶数组长度是2的幂，阈值大小为数组长度和负载因子的乘积，当HashMap中的键值对超过阈值的时候，就会进行扩容，resize()扩容主要是做三件事情：

  • 计算新桶数组的容量和新阈值
  • 根据计算出来的新阈值创建新的桶数组，桶数组table也是在这里初始化的。
  • 将键值对节点重新映射到新的桶数组里面，如果节点是TreeNode类型的话，就转换为红黑树，普通节点就还是原顺序进行分组。

总结

很多细节还没有补充完毕，这里简单说下HashMap的特点：

  • hashMap不是线程安全的，如果想要线程安全的可以去用ConcurrentHashMap。
  • hashMap不保证键值对顺序，并且允许空值null。
  • hashMap的默认容量是16，每次扩容是都是原来的两倍，而且HashMap 总是使用 2 的幂作为哈希表的大小。
  • 加载因子不能太大或者太小，太大会导致查找元素效率低，太小的话会导致数组的利用率低，存放的数据就会很分散。

如果想要更深一步的去了解hashMap可以去看下红黑树的特性。

操作系统复习-进程与线程

进程基本概念

多道程序设计

允许多个进程同时进入内存并运行，目的是为了提高系统效率。

并发环境

一段时间间隔内，但处理器上有两个或者两个以上的程序处于开始运行但尚未结束的状态，并且次序不是事先确定的

并发程序

在并发环境中执行的程序。

进程的定义

进程是具有独立功能的程序关于某个数据集合上的一次运动活动，是系统进行资源分配和调度的独立单元，进程是对CPU的抽象。

进程的特点

  • 程序的一次执行过程。
  • 是正在运行程序的抽象
  • 将一个CPU虚拟化为多个CPU
  • 系统资源以进程为单元进行分配，比如内存，文件等等，每个进程都具有独立的地址空间。
  • 操作系统将CPU调度给需要的进程。

进程控制块(进程描述符/进程属性)PCB

  • 操作系统用于管理控制进程的一个专门数据结构。
  • 记录进程的各种属性，描述进程的动态变化过程。
  • PCB和进程一一对应，是系统感知进程存在的唯一标志。
  • 进程表：所有进程的PCB集合。

PCB保存的信息

  • 进程描述信息
  • 进程控制信息
  • 进程所持有的资源和使用情况
  • CPU现场信息

进程描述信息

  • 进程标识符(PID)唯一，通常是一个整数。
  • 进程名，通常基于可执行文件名，不唯一。
  • 用户标识符
  • 进程组关系

进程控制信息

  • 当前进程状态
  • 进程优先级

进程状态

进程的三种基本状态

  • 运行态: 占用CPU，并在CPU上执行。
  • 就绪态: 已经具有运行条件，但是因为没有空间CPU，而暂时不能运行。
  • 等待态(阻塞态/封锁态/睡眠态): 因等待一事件而暂时不能运行。

进程状态改变

  • 就绪态->运行态
  • 运行态->就绪态
  • 运行态->等待态

处于就绪态和运行态的进程都是可以运行的，只是就绪态暂时没有CPU而已，处于等待态的进程不能运行，就算CPU空闲也不可以。

当调度程序选择一个新的进程运行的时候，就绪态转换为运行态，处于运行态上的进程用完时间片了或者遇到一个高优先级的进程进入到了就绪态，那这个时候运行态上的进程就要切换为就绪态。

当一个进程等待某个事件发生的时候，就会从运行态进入到等待态，当事件发生的时候，就会从等待太回到就绪态。

进程的其他状态

  • 创建态
    • 已经完成了一个进程所必要的工作
      • PID
      • PCB
    • 但尚未同意执行该进程，因为资源有限。
  • 终止态
    • 终止执行后，进程进入该状态。
    • 可以完成一些数据统计工作。
    • 资源回收
  • 挂起态
    • 用于调节负载
    • 进程不占用内存空间，其进程映像交换到磁盘上。

进程和程序的区别

  • 进程能够更准确地刻画并发，而程序不能。
  • 程序是静态的，进程是动态的。
  • 进程有生命周期，相对程序来说比较短暂，而程序是相对长久的。
  • 一个程序可以对应多个进程
  • 进程具有创建其他进程的功能

举个例子，食谱可以看成一个程序，进程是食谱上的菜。

进程地址空间

每个进程有相对独立的地址空间，彼此之间隔离。

进程映像

对进程执行活动全过程的静态描述，由进程地址空间内容硬件寄存器内容以及该进程相关的内核数据结构，内核栈组成。

  • 用户相关：进程地址空间(包括代码段、数据段、堆和栈、共享库等等)
  • 寄存器相关：程序计数器、指令寄存器、程序状态，栈指针、通用寄存器等的值。
  • 内核相关
    • 静态部分：PCB以及各种资源数据结构。
    • 动态部分：内核栈(不同进程在进入内核使用不同的内核栈)

上下文切换

将CPU硬件状态从一个进程切换到另一个进程的过程称为上下文切换，进程运行的时候，其硬件状态保存在CPU上的寄存器中，寄存器包括以下几种：

  • 程序计数器
  • 程序状态寄存器
  • 栈指针
  • 通用寄存器
  • 其他控制寄存器

进程不运行的时候，这些寄存器的值就要保存在进程控制块PCB中，当操作系统要运行一个新的进程时，将PCB中的相关值送到对应的寄存器中。

线程

为什么需要引用线程？因为如下原因：

  • 应用的需要
  • 开销的考虑
  • 性能的考虑

开销的考虑

线程相对于进程来说的开销优势是：创建一个新的线程花费时间少，撤销线程也是一样的，两个线程的切换花费时间也少，同时线程之间的通信不需要使用到内核，因为同一个进程内都是共享内存和文件的。

线程的基本概念

进程的两个概念

  • 资源的拥有者
  • CPU调度单位

线程是进程中的一个运行实体，是CPU的调度单位，有时候将线程称为轻量级线程，在同一个进程中添加了多个执行序列。

线程属性

  • 有标识符ID
  • 有状态以及状态转换
  • 不运行的时候需要保存上下文，有上下文环境：程序计数器等寄存器。
  • 有自己的栈和栈指针。
  • 同一个进程中不同的线程共享地址空间和其他资源。
  • 可以创建和撤销进程，因为程序一开始以一个单线程运行。

线程机制的实现

  • 用户级线程
  • 核心级线程
  • 混合-两者结合方法

用户级线程

  • 在用户空间建立线程库，提供一组管理线程的过程。
  • 运行时系统：完成线程的管理工作(操作、线程表)
  • 内核管理的还是进程，不知道线程的存在。
  • 线程的切换不需要内核态特权

UNIX就是实现了用户级线程。

用户级线程总结

优点

  • 线程切换快
  • 调度算法是应用程序特定的
  • 用户级线程可以运行在任何操作系统上(只需要操作系统实现了线程库)

缺点

  • 内核只将处理器分配给进程，同一个进程中的两个线程不能同时运行在两个处理器上，因为进程是CPU的调度单位。
  • 大多数系统调用都是阻塞的，所有内核阻塞进程，进程中的所有线程都被阻塞。

核心线程

  • 内核管理所有线程，并向应用程序提供API接口。
  • 内核维护进程和线程的上下文。
  • 线程的切换需要内核支持
  • 以线程为基础进行调度

比如Windows。

混合模型

  • 线程创建在用户控件完成
  • 线程调度等在核心态完成
  • 例子：Solaris

多个用户级线程，多路复用。

总结

进程特性

  • 并发性：任何进程都可以和其他进程一起向前推进。
  • 动态性：进程是正在执行程序的实例。
    • 进程是动态产生，动态消亡的。
    • 进程在其生命周期内，在三种基本状态之间转换。
  • 独立性：进程是资源分配的一个独立单位，比如每个进程的地址空间相互独立。
  • 交互性：进程在执行过程中可能和其他进程产生直接或间接关系。
  • 异步性：每个进程都以相对独立的，不可预知的速度向前推进。
  • 进程映像：程序+数据+栈(用户栈，内核栈)+PCB。

MapRedcude论文笔记

MapReduce

  • MapReduce是什么？
  • MapReduce如何实现？
  • 什么是Map？
  • 什么是Reduce？

MapReduce是一个编程模型，同时也是一个能够处理和生成超大数据集的算法模型，用户首先创建一个Map函数处理一个基于KV数据集合，输出中间基于KV的数据集合。

然后在创建一个Reduce函数用来合并所有具有相同中间key值的中间value值。

MapReduce的产生背景Google程序员为了处理海量的原始数据而实现的，比如文档抓取，Web请求日志等等，也有为了处理计算机各种类型的衍生数据，比如倒排索引，Web文档的图结构的各种表示形式等。

MapReduce搭建配置

论文中MapReduce的程序都是能够在大量的普通配置的计算机上实现并行处理的，MapReduce只关心以下几点：

  • 如何分割数据
  • 如何在大量的计算机组成的集群上进行调度
  • 集群中计算机的错误处理
  • 管理集群中计算机的必要通信

一个典型的MapReude算法模型都是几千台机器组成，处理以TB计算的数据。

MapReduce解决了什么？

在没有MapReduce之前，如果输入的数据量特别大，比如1TB左右，又想要在短时间内处理，或者说是能够接受的时间内完成，考虑到单机的性能，就需要将这些计算分布在其他主机上，但是这就涉及到了如何并行计算，如何分发数据，如何处理错误，这些问题综合在一起就不再是简单的输入输出处理了。

Map函数

Map是接受输入并对输入进行处理的一个函数，举个例子，输入的是一辆汽车，通过map函数，最后输出的是这辆汽车原来的零件。

Reduce函数

如果Map是对汽车进行拆解，那Reduce就是对拆解完之后的零件，进行拼装，那能不能还原回去呢？

单词统计的例子

打个比方，有一个待输入的文本，首先通过Map函数，对文本进行处理，对每个单词出现的次数进行统计，然后输出成一个KV数据集合，Reduce把Map函数产生的每一个特定的词的次数累加起来。

Map函数处理：

(From,1)

(fairest,1)

(creatures,1)

(we,1)

(desire,1)

(increase,1)

MapReduce应用例子

  • Distributed Grep
  • Count of URL Access Frequency
  • Reverse Web-Link Graph
  • Term-Vector per Host
  • Inverted Index
  • Distributed Sort

MapReduce实现

MapReduce模型的实现有很多种，具体实现还是要看使用场景，比如一种实现是适用于小型的共享内存方法的机器，另外一种实现方式则适用于大型 NUMA 架构的多处理器的主机，而有的实现方式更适合大型的网络连接集群。

如果是为了测试，可以使用docker来模拟MapReduce实现。

执行过程

通过将Map调用的输入数据自动分割为M个数据片段集合，Map调用被分布到多台机器上执行，输入的数据片段能够在不同的机器上并行处理，使用分区函数Map调用产生的中间key值被分成R个不同分区，Reduce调用也被分不到多态机器上执行，分区数量R和分数函数都是通过用户来指定。

  1. 将输入的文件分成M数据片段，每个数据片段大小一般在16MB到64MB，可以通过可选参数来控制每个数据片段大小，然后通过用户程序在机器创建大量的程序副本。
  2. 程序副本由两个程序组成，master和worker，通过master分配任务，由M个Map任务和R个Reduce任务将会被分配，master将一个Map任务或者Reduce任务分配给一个空闲的worker。
  3. 如果worker是被分配到了map任务，那它就会读取相关输入数据片段，从输入的数据片段中解析出KV集合，然后把KV集合传递给用户自定义的Map函数，再通过Map函数生成输出的中间KV集合缓存在内存中。
  4. 缓存中的KV集合通过分区函数分成R个区域，之后周期性的写入到本地磁盘上，缓存中KV集合在本地磁盘上的存储位置将会传回给master，再由master把这些存储位置传送给Reduce worker。
  5. 当Reduce woker程序接收到master程序发送过来的数据存储位置信息之后，使用RPC从Map worker所在的主机上的磁盘上读取缓存数据，当Reduce worker读取了所有的中间数据之后，通过对key进行排序，让具有相同的key值聚合在一起。因为许多不同的key值会映射到相同的reduce任务上，所有就需要进行排序，如果中间数据量太大，没有办法在内存中排序的话，就需要使用外部排序。
  6. 当reduce worker程序遍历排序之后的中间数据，对每一个唯一的中间key值，reduce worker程序会将这个key值和它相关的中间value值得集合传递给用户自定义得Reduce函数，Redcude函数得输出被追加到所属分区得输出文件。
  7. 当所有得Map和Reduce任务都完成之后，master唤醒用户程序，在这个时候，在用户程序里对MapReduce 调用才返回。

为什么要把输入的文件分成16MB到64MB的数据片段呢？这是因为如果要把输入数据分成相同的数据片段大小并不是一件容易的事情，有的输入文件大有的输入文件小，虽然有些线程会比其他线程更早完成任务，通过调度算法去完成下一个任务，但是这样的话，最后完成的时间取决于最大的文件。

而如果划分为固定大小的数据片段，然后分到各个进程里面，这样一来的话， 即使有一些进程可以处理更多的数据，我们也可以分配更多的数据给它。

Master 数据结构

Master的数据结构记录着每个Map和Reduce任务状态(空间、工作或完成)，以及Worker机器的标识(非空闲任务机器)的标识。

对于每个已经完成的Map任务，master存储了Map任务产生的R个中间文件存储区域大小和位置，当Map任务完成的时候，Map接收到位置和大小更新信息，这些信息被逐步递增的推给那些正在工作的Reduce任务。

容错性(Fault Tolerance)

worker失败

master会周期性的ping一圈集群中的每个worker，如果在一个规定的时间内worker没有发出响应信息，master就会把这个worker标记为失效。

失效worker上所有的Map任务会被重置为初始状态，之后把这些任务安排给其他的worker，同样的，如果是Redcue任务也会被重新置为空闲状态，等待重新调度。

master失败

master失败就比较麻烦，因为MapReduce只有一个master进程，如果它失败了，就会终止MapReduce运算，客户可以检查到这个状态，并且可以根据需要重新执行MapReduce操作。

存储位置

MapReduce尽量把输入数据(由GFS管理)存储在集群中机器的本地磁盘上来节省网络带宽，GFS把每个文件按64MB一个Blokc分隔，每个Block保存在多台机器上，环境就存放了多份拷贝(一般是3个拷贝)。MapReduce的maser在调度Map任务的时候会考虑输入文件的位置信息，尽量将一个Map任务调度在包含相关输入数据拷贝的机器上执行。

如果失败了，那master就会尝试在保存有输入数据拷贝的机器附近的机器上执行 Map 任务。
